# RE-Environment-Onto (REO)

**Version:** 1.0.0  
**Language:** OWL 2 + SHACL (Turtle syntax)  

## Description

REO is the formal ontology of the **EE Modelisation** project (*Environment-Centric Requirements Engineering*). It encodes the seven dimensions of the socio-technical environment:

```
E = <A, G, D, C, X, U, I>
```

- **A** — Actors (Humans, Systems, Organizations)
- **G** — Goals (Strategic & Operational, with AND/OR hierarchy)
- **D** — Domain (Concepts, Properties, Invariants)
- **C** — Constraints (Regulatory, Business, Technical, Quality)
- **X** — Contexts (Operational situations)
- **U** — Uncertainties (Unknowns, Assumptions, Ambiguities, Risks)
- **I** — Interactions (Data flows, Communications, Service calls)

## Files

### `ontology/reo-core.ttl`
Main OWL schema defining:
- 27 classes (Actor, Goal, DomainConcept, Constraint, Context, Uncertainty, Interaction...)
- 57 properties (datatype & object properties)
- Class hierarchies and disjointness
- Six canonical OWL cardinality restrictions:
  - at least one `authorityLevel` for every Actor;
  - exactly one `hasSource` for every Interaction;
  - at least one `hasAuthoritativeText` for every RegulatoryConstraint;
  - at least one `impactLevel` and one `status` for every Uncertainty.
- SKOS multilingual documentation (EN)

The restrictions are maintained only in `reo-core.ttl`, which is the single
source of truth for the OWL schema. No separate axiom copy is distributed.

### `ontology/reo-shapes.ttl`
SHACL shapes for instance validation:
- 12 NodeShapes targeting the main classes
- Cardinality, type, and enumerated value constraints
- Referential integrity (every reference must exist)
- Acyclicity detection in Goal hierarchy (via SPARQL)
- Warning generation for unresolved uncertainties blocking Goals

### `examples/medical-emergency/hospital.ttl`
Complete instance illustrating the "Emergency Prescription" case from the EE Modelisation document:
- 3 actors: Physician, Pharmacist, EHR_System
- 5 goals: G001 (Strategic) → G002-G005 (Operational/leaf)
- 2 domain concepts: Medication, Patient (with properties and invariants)
- 2 constraints: HIPAA Privacy Rule, FDA Prescription Rules
- 1 context: Emergency Department
- **1 uncertainty: Drug DB Update Frequency (U-001) — status "unvalidated"**
- 2 interactions: Physician→EHR (Patient read), EHR→Pharmacy (Prescription forward)

> **Note:** This instance generates a SHACL **warning** (not a violation) due to the unresolved uncertainty U-001. This is the expected behavior of the REO model: an unresolved uncertainty signals conditional requirements without blocking validation.

### `test-cases/`
Systematic regression tests for SHACL shape validation:

| Test | File | Expected Result |
|------|------|-----------------|
| 01 — Success | `01-success/hospital-success.ttl` | `Conforms: True` (no warnings) |
| 02 — Success with Warning | `02-success-with-warning/hospital-warning.ttl` | `Conforms: True` (warning U-001) |
| 03 — Violations | `03-violations/hospital-violations.ttl` | `Conforms: False` (20+ violations) |

### `scripts/validate_syntax.py`
Validates the Turtle syntax of all `.ttl` files in the project using `rdflib`.

### `scripts/artifact_metrics.py`
Regenerates `artifact_metrics.json` directly from the six Turtle files. The
current corpus contains 1,492 triples: 433 in `reo-core.ttl`, 354 in
`reo-shapes.ttl`, 207 in the medical instance, and 498 across the three
regression fixtures.

## Continuous Integration

This project uses **GitHub Actions** for automated validation:

- **Pinned environment:** Python 3.11.15, RDFLib 7.6.0, and pySHACL 0.40.0
- **Syntax check:** All six `.ttl` files are parsed with RDFLib
- **Artifact metrics:** `artifact_metrics.json` is regenerated from the six Turtle files
- **SHACL validation:** the main instance and all three fixtures are regenerated
- **Result assertions:** expected conformance, warning, and violation counts are checked
- **Derivation contract:** 18 state scenarios and four impact queries are executed
- **Auto-commit:** generated reports, `evaluation_results.csv`, and `artifact_metrics.json` are updated on main/master pushes
- **Artifacts:** all generated reports, the CSV, and the metrics JSON are uploaded as workflow artifacts

See `.github/workflows/reo-validation.yml` for the full pipeline.

## Usage

### With Apache Jena
```bash
# Syntax validation
riot --validate ontology/reo-core.ttl
riot --validate ontology/reo-shapes.ttl
riot --validate examples/medical-emergency/hospital.ttl

# OWL reasoning
riot --rdfs=ontology/reo-core.ttl examples/medical-emergency/hospital.ttl
```

### With pySHACL
```bash
# Validate main instance
pyshacl -s ontology/reo-shapes.ttl   -e ontology/reo-core.ttl   -i rdfs   -f human   examples/medical-emergency/hospital.ttl

# Validate test cases
pyshacl -s ontology/reo-shapes.ttl   -e ontology/reo-core.ttl   -i rdfs   -f human   test-cases/01-success/hospital-success.ttl

pyshacl -s ontology/reo-shapes.ttl   -e ontology/reo-core.ttl   -i rdfs   -f human   test-cases/02-success-with-warning/hospital-warning.ttl

pyshacl -s ontology/reo-shapes.ttl   -e ontology/reo-core.ttl   -i rdfs   -f human   test-cases/03-violations/hospital-violations.ttl
```

### With Protégé
1. Open `reo-core.ttl` (Active OWL profile: OWL 2 DL). The six OWL restrictions are already included.
2. Load `examples/medical-emergency/hospital.ttl` into a separate ontology or as individuals.
3. Use the Reasoner (HermiT / Pellet) to inspect the OWL restrictions and inferred types.
4. Use `reo-shapes.ttl` with a SHACL engine for closed-world validation.

## Reference Derivation Checker (`prototype.py`)

While the SHACL layer decides whether an environment *model* conforms, the
reference checker decides the requirement state assigned to a candidate. It
reads the same `examples/medical-emergency/hospital.ttl` instance, creates a
seed slice, completes it with candidate-specific validation evidence, and
applies the contract deterministically. No LLM is invoked.

```bash
python scripts/artifact_metrics.py
python prototype.py
python scripts/assert_derivation_results.py
# Scenario checks: 18/18 passed
# Impact queries : 4/4 passed
# Baseline status: CONDITIONAL; trace size: 9
# Results written: evaluation_results.csv
```

The checker implements:

- `slice_env(env, seed, context)` — the guarded goal-seeded relevance policy;
- `validation_closure(env, S0, q, context)` — candidate-specific slice closure;
- `validate(env, S0, q, context)` — the requirement-state contract;
- `requirements_of(e, records)` — direct trace-relative change impact.

The validation-state priority is:

```text
rejected
> pending_review
> conditional
> accepted_with_reservations
> accepted
```

`not_derived` is returned before candidate validation when no active anchoring
slice exists. Applicable hard constraints must be present and discharged by an
explicit registered checker; otherwise the candidate becomes `pending_review`
rather than being accepted by default.

The generated CSV contains 18 scenarios: 2 accepted, 1 accepted with
reservations, 2 conditional, 3 pending review, 9 rejected, and 1 not derived.
It also contains four direct impact queries (three hits and one miss). See
`test-cases/derivation-scenarios.md` for the complete catalogue.
